# Use Cases

The Memori MCP server works with any AI agent that supports MCP — not just IDE agents. Below are three common integration patterns: Slack bots, Notion AI assistants, and LangChain agents.

## Slack

A Slack bot backed by an LLM can use Memori MCP to remember each user's preferences, communication style, and team context across every conversation — no matter which channel or thread the interaction happens in.

### How It Works

| Header | Recommended Value |
|---|---|
| `X-Memori-Entity-Id` | Slack user ID (e.g. `U04ABCDEF`) |
| `X-Memori-Process-Id` | Workspace or bot name (e.g. `slack_support_bot`) |

On each incoming message, the bot:

1. Calls `recall` with the user's message as the query
2. Injects the returned memories into the system prompt or context window
3. Generates and sends the response
4. Calls `advanced_augmentation` with the full user/assistant turn

### What Gets Remembered

Slack-specific facts that are worth persisting:

- Preferred response length and tone ("keep it brief", "use bullet points")
- Timezone and working hours
- Recurring responsibilities or project ownership ("Alice owns the on-call rotation")
- Team conventions ("all deploys go through #releases before 5pm EST")
- Preferred escalation paths or tool preferences

### What to Skip

- Message contents that contain secrets, tokens, or PII
- Incident timelines or live status updates (ephemeral by nature)
- One-off clarifications that won't matter next week

### Example: Support Bot

A user messages your support bot: _"I always prefer seeing raw SQL over ORM examples."_

The bot stores this via `advanced_augmentation`. The next time the same user asks a database question — in any channel, days later — `recall` surfaces the preference and the response uses raw SQL automatically.

### MCP Config for a Slack Bot

If your bot runs as a Node.js or Python service, set the headers dynamically per-request using the Slack user ID from the incoming event payload:

```ts
const memoriHeaders = {
  "X-Memori-API-Key": process.env.MEMORI_API_KEY,
  "X-Memori-Entity-Id": slackEvent.user,         // e.g. "U04ABCDEF"
  "X-Memori-Process-Id": "slack_support_bot",
};
```

Pass these headers in every MCP tool call. `session_id` is derived automatically from `entity_id` and the current UTC hour.

## Notion

Notion AI assistants and custom Notion integrations can use Memori MCP to maintain persistent context about a user's writing style, project structure, and long-lived preferences — so every AI interaction feels consistent regardless of which page or database is open.

### How It Works

| Header | Recommended Value |
|---|---|
| `X-Memori-Entity-Id` | Notion user ID (e.g. `notion_user_abc123`) |
| `X-Memori-Process-Id` | Workspace name or integration name (e.g. `notion_writing_assistant`) |

On each AI-assisted action (e.g. "Help me write this section"), the assistant:

1. Calls `recall` with the user's prompt or selected text as the query
2. Uses the returned memories to personalize tone, structure, and terminology
3. Generates the content or response
4. Calls `advanced_augmentation` with the prompt and generated output

### What Gets Remembered

Notion-specific facts worth persisting:

- Writing tone and style ("always use active voice", "avoid jargon")
- Document structure preferences ("lead with the TL;DR", "use H3s not H2s")
- Project-level terminology and definitions ("'Sprout' refers to our beta feature set")
- Recurring content patterns ("meeting notes always follow this template")
- Preferred formatting ("use tables over bullet lists for comparisons")

### What to Skip

- Draft content in progress (ephemeral until the user confirms it)
- Temporary reminders or one-off notes
- Page titles or IDs (structural metadata, not durable preferences)

### Example: Writing Assistant

A user asks the assistant: _"Help me draft the intro — I always write in second person and keep intros under 100 words."_

The assistant stores both constraints via `advanced_augmentation`. On the next page — a week later — when the user says _"Write an intro for this"_, `recall` surfaces the style rules and the assistant applies them without being told again.

### MCP Config for a Notion Integration

Set the entity and process IDs from the Notion API user object:

```ts
const memoriHeaders = {
  "X-Memori-API-Key": process.env.MEMORI_API_KEY,
  "X-Memori-Entity-Id": notionUser.id,              // e.g. "notion_user_abc123"
  "X-Memori-Process-Id": "notion_writing_assistant",
};
```

Use `process_id` to isolate memories by integration or workspace so preferences from a personal workspace don't bleed into a team workspace.

## LangChain

LangChain agents can use Memori MCP tools via the [`langchain-mcp-adapters`](https://github.com/langchain-ai/langchain-mcp-adapters) package. This gives any LangChain or LangGraph agent access to `recall` and `advanced_augmentation` without any custom HTTP wiring — the adapter handles tool discovery and invocation automatically.

### How It Works

| Header | Recommended Value |
|---|---|
| `X-Memori-Entity-Id` | End-user ID from your app's auth layer (e.g. `user_123`) |
| `X-Memori-Process-Id` | Agent or chain name (e.g. `my_langchain_agent`) |

On each user request, the agent:

1. Calls `recall` with the user's message as the query
2. Uses the returned memories to ground and personalize the response
3. Generates and returns the response
4. Calls `advanced_augmentation` with the full user/assistant turn

### What Gets Remembered

LangChain-specific facts worth persisting:

- User preferences for response format, verbosity, or language
- Domain knowledge the agent should always apply ("our data pipeline runs on Airflow")
- Long-lived project or workflow context ("always validate against the staging environment first")
- Recurring constraints or coding standards ("use async/await, never callbacks")
- Model or chain preferences the user has expressed ("keep tool calls to a minimum")

### What to Skip

- Intermediate chain steps or tool outputs (ephemeral reasoning, not durable facts)
- One-off parameters passed in a single request
- LLM usage stats, token counts, or latency measurements

### Example: Personalized Code Agent

A user tells the agent: _"I always want type hints and docstrings in any Python code you write."_

The agent stores both rules via `advanced_augmentation`. In a later session, when the user asks it to write a utility function, `recall` surfaces the style rules and the agent applies them without being re-instructed.

### MCP Config for a LangChain Agent

Install the adapter package:

```bash
pip install langchain-mcp-adapters
```

Use `MultiServerMCPClient` with `streamable_http` transport. Instantiate it per request so `entity_id` is always scoped to the current user:

```python
import os
from langchain_mcp_adapters.client import MultiServerMCPClient
from langgraph.prebuilt import create_react_agent
from langchain_anthropic import ChatAnthropic

model = ChatAnthropic(model="claude-sonnet-4-6")

async def handle_request(user_id: str, user_message: str) -> str:
    async with MultiServerMCPClient(
        {
            "memori": {
                "url": "https://api.memorilabs.ai/mcp/",
                "transport": "streamable_http",
                "headers": {
                    "X-Memori-API-Key": os.environ["MEMORI_API_KEY"],
                    "X-Memori-Entity-Id": user_id,       # e.g. "user_123"
                    "X-Memori-Process-Id": "my_langchain_agent",
                },
            }
        }
    ) as client:
        tools = client.get_tools()
        agent = create_react_agent(model, tools)
        result = await agent.ainvoke(
            {"messages": [{"role": "user", "content": user_message}]}
        )
        return result["messages"][-1].content
```

`session_id` is derived automatically from `entity_id` and the current UTC hour. Because `MultiServerMCPClient` is used as an async context manager, tool connections are opened and closed cleanly on each request.

## Shared Patterns

All three use cases follow the same recall-then-augment loop described in the [MCP Overview](/docs/memori-cloud/mcp/overview):

1. `recall` → fetch relevant memories before generating a response
2. Generate response using returned context
3. `advanced_augmentation` → store durable facts after responding

The key difference from IDE agents is that `entity_id` is set dynamically per request (from the Slack user ID, Notion user ID, or your app's auth layer) rather than once in a static config file.

<Note>
  For multi-tenant applications (bots serving many users), never hard-code `entity_id`. Always derive it from the authenticated user's ID in the incoming request so memories remain user-scoped and isolated.
</Note>
